# Devlog Rule

## 원칙

- 모든 작업은 **계획 먼저 → 승인 후 실행 → 실시간 기록** 순서로 진행한다.
- 사용자의 승인 없이 코드 변경을 시작하지 않는다.

## 워크플로우

### 1단계: 작업 계획 작성 (작업 전)

작업 요청을 받으면 코드를 수정하기 전에 `docs/devlog/YYYY-MM-DD-{number}-{description}.md` 파일을 먼저 생성한다.

아래 형식으로 계획을 작성한다:

```markdown
## [HH:MM] 작업 계획

### 요청 사항
- 사용자가 요청한 내용 요약

### 구현 계획
- [ ] 할 일 1 — 변경 대상 파일, 방법 간단히
- [ ] 할 일 2
- [ ] 할 일 3

### 예상 변경 파일
- `path/to/file.py` — 변경 이유
- `path/to/file2.py` — 변경 이유

### 의사결정
- 선택한 방식: 설명
- 대안: 왜 선택하지 않았는지

### 리스크/주의사항
- 있으면 기록
```

### 2단계: 사용자 승인 대기

계획 작성 후 반드시 사용자에게 리뷰를 요청한다:

> "위 계획을 확인해주세요. 수정할 부분이 있으면 말씀해주세요."

- 사용자가 승인하거나 수정 요청을 할 때까지 **코드 변경을 하지 않는다**
- 수정 요청이 오면 계획을 업데이트하고 다시 승인을 요청한다

### 3단계: 작업 실행 및 실시간 업데이트

승인을 받으면 작업을 진행하면서 devlog를 실시간으로 업데이트한다:

- 각 할 일 완료 시 체크박스를 `[x]`로 변경
- 예상과 다르게 진행된 부분이 있으면 바로 기록
- 추가로 변경된 파일이 있으면 목록에 추가

### 4단계: 작업 완료 정리

모든 작업이 끝나면 devlog 하단에 완료 요약을 추가한다:

```markdown
## [HH:MM] 작업 완료

### 결과
- 완료된 작업 요약

### 실제 변경 파일
- `path/to/file.py` — 실제 변경 내용

### 계획 대비 변경점
- 계획과 달라진 부분이 있으면 기록

### 다음 할 일
- 이어서 해야 할 작업
- 발견된 이슈나 개선점

### 이슈/블로커
- 해결 못한 문제가 있으면 기록
```

## 규칙

1. **각 작업마다 별도 파일**로 작성한다 (같은 날 여러 작업이 있어도 파일 분리)
2. **이전 devlog가 있으면** 반드시 읽고 맥락을 이어간다
3. **간결하게** 작성하되, 다음 세션에서 이어 작업할 수 있을 만큼의 정보는 포함한다
4. **devlog 작성 자체**는 사용자 승인 없이 진행한다 (코드 변경만 승인 필요)

## 파일명 규칙

- 형식: `YYYY-MM-DD-{number}-{description}.md`
- 예시: `2026-02-04-01-cloud-run-startup-fix.md`
- 위치: `docs/devlog/`
- `{number}`: 같은 날짜 내 작업 순서 (01, 02, 03, ...)
- `{description}`: 작업 내용을 간략히 표현 (kebab-case)
